IDEA 1 / 3Stack — plates ka meenar

Java/C# program ki memory ke do bare hisse hote hain. Pehla: stack.

Shaadi wale ghar mein plates ka meenar socho. Nayi plate hamesha upar rakhi jati hai, aur uthti bhi hamesha upar se hai. Program mein har dafa koi method call hoti hai, uske liye ek nayi "plate" upar rakh di jati hai — is plate pe us method ki saari local cheezein hoti hain: int x = 5, loop ke counters, parameters. Method khatam? Poori plate utha li jati hai — us pe rakhi har cheez ek saath ghayab. Koi safai wala nahi chahiye, koi dhoondhna nahi — bas ek harkat mein saaf.

main() chali          →  plate 1 rakhi   [ main ke locals ]
main ne openAccount() ko bulaya
                      →  plate 2 UPAR    [ openAccount ke locals: x, c ]
openAccount ne validate() ko bulaya
                      →  plate 3 UPAR    [ validate ke locals ]
validate() khatam     →  plate 3 utha li — uske sab locals gone
openAccount() khatam  →  plate 2 utha li — x aur c gone

Isi liye stack bohot tez hai — rakhna aur uthana bas "meenar ki aakhri manzil" ka hisaab badalna hai. Lekin ek shart hai: plate pe wohi cheez reh sakti hai jo method ke saath khatam ho jaye. Aur yahin se doosra sawal nikalta hai...

IDEA 2 / 3Heap — godown aur parchi

Fatima ka account object method khatam hone ke BAAD bhi zinda rehna chahiye — warna bank kis kaam ka? Aisi cheezon ke liye doosra hissa hai: heap — ek bara godown jahan objects rehte hain.

Jab tum likhte ho Customer c = new Customer(), do cheezein hoti hain:

  • new Customer() → asal object godown (heap) mein rakha jata hai
  • c → tumhari plate (stack) pe sirf ek parchi rakhi jati hai jis pe godown ka pata likha hai — isko reference kehte hain
Reference = object ka pata. Courier ki receipt socho: parcel (object) warehouse (heap) mein hai, tumhare haath mein sirf receipt (reference) hai. Do log ek hi parcel ki receipt rakh sakte hain — object EK hai, parchiyan do. (Isi liye Integer wale sawal mein == "same pata?" poochta hai, "same qeemat?" nahi.)

Ab kal wale W1 ko dobara dekho — ab yeh ratta nahi, samajh hai: x (aam number) plate pe, c ki parchi plate pe, asal Customer object godown mein.

IDEA 3 / 3Kaun kab marta hai — aur Danish ka muqadma

Stack ka hisaab saaf hai: method khatam = plate gone. Lekin godown ka? Objects wahan tab tak zinda hain jab tak... kya?

Usool yaad kar lo: object tab tak zinda hai jab tak us tak pahunchne wali koi parchi maujood hai. Jab aakhri parchi bhi ghayab ho jaye (plate uth gayi, ya kisi ne parchi pe naya pata likh diya), to object lawaris ho jata hai — us tak koi pahunch hi nahi sakta. Aise lawaris objects ko JVM/CLR ka safai-wala utha leta hai — isko GC (Garbage Collector) kehte hain. (GC ki apni kahani bari dilchasp hai — woh Day 4 pe poori milegi. Aaj bas itna: woh SIRF lawaris cheezein uthata hai.)

Ab adalat lagti hai. Danish ka code pesh ho:

❌ MULZIM: Danish ka failed-login tracker 
public class LoginSecurity {
    // "static" = yeh Map kisi ek object ki nahi,
    // poori CLASS ki cheez hai — program bhar zinda
    static Map<String, Integer> failedAttempts = new HashMap<>();

    public static void recordFailure(String cnic) {
        failedAttempts.put(cnic, failedAttempts.getOrDefault(cnic, 0) + 1);
        // ...aur NIKAALTA kabhi nahi
    }
}

Gawahi suno. Har failed login pe ek entry godown mein gayi. Us entry ki parchi kis ke paas hai? static failedAttempts Map ke paas. Aur static cheez kab marti hai? Kabhi nahi — woh program ke saath hi khatam hoti hai, uski parchi kabhi nahi phatti. To har entry hamesha "zinda" — GC aata raha, dekhta raha, "yeh to sab claimed hain", khaali haath wapas. Godown bharta gaya. GC bar bar bhaagta raha (isi liye pehle slow hua — W2 ka jawab!), aur aakhir mein jagah hi khatam: OutOfMemoryError.

Faisla: Map begunah, static begunah — jurm yeh tha ke jama karte gaye aur nikaala kabhi nahi. Ab saza nahi, islaah:

✅ ISLAAH: hadd bandho, purana nikaalo 
// Fix ka usool bas do lafz: BOUND + EXPIRE
// (1) size ki hadd — is se zyada kabhi na ho
// (2) purani entries ki miyaad — kaam khatam, entry khatam
if (failedAttempts.size() > 10_000) {
    failedAttempts.clear();   // sab se sada ilaaj (behtar: purani nikaalo)
}

Asal duniya mein is kaam ke liye ready-made "bounded cache" libraries hoti hain jo size aur miyaad dono khud sambhalti hain (Java mein mashhoor: Caffeine) — Lab wale din hum yehi Danish ke code pe lagayenge. Aaj usool pakka karo: jo cheez jama hoti hai, uske nikalne ka rasta pehle likho.

Danish
"Sir lekin... Java mein to GC hai na. Mujhe bataya gaya tha Java mein memory leak ho hi nahi sakti."
Tum
"GC lawaris cheezein uthata hai, Danish. Tumne to har entry ko static parchi se bandh ke rakha tha — woh lawaris kabhi hui hi nahi. Java mein leak hoti hai — bas uski shakal 'bhoola hua reference' hoti hai. Yeh jumla likh lo, interview mein sona hai."

STACK 2 • .NET CORE / C#Danish ka juram C# mein — aur framework wala ilaaj

TICH-KOLEY ka aadha backend .NET Core mein hai (tumhari asli zindagi ki tarah). Wahan bilkul yehi memory model hai — stack, heap, GC, static ka wohi khatra. Danish ka juram C# mein aise dikhta:

❌ WOHI JURAM, C# KI WARDI MEIN 
public static class LoginSecurity
{
    // static + kabhi remove nahi = wohi amar parchi
    static Dictionary<string, int> failedAttempts = new();

    public static void RecordFailure(string cnic)
        => failedAttempts[cnic] =
             failedAttempts.GetValueOrDefault(cnic) + 1;
}

Lekin .NET Core ke paas is kaam ke liye ek banaa-banaya, parha-likha auzaar hai: IMemoryCache — ek in-memory cache jo bound + expire dono khud sambhalta hai. Dekho, line-ba-line:

✅ .NET CORE KA SAHI TAREEQA — IMemoryCache 
// Program.cs — cache register karo, HADD ke saath:
builder.Services.AddMemoryCache(o => o.SizeLimit = 10_000);
//                                    ↑ BOUND: is se zyada entries kabhi nahi

// service mein:
public void RecordFailure(string cnic)
{
    int attempts = _cache.Get<int>(cnic);   // pichli ginti (na ho to 0)
    _cache.Set(cnic, attempts + 1, new MemoryCacheEntryOptions()
        .SetSize(1)                              // har entry ka "wazan" = 1
        .SetSlidingExpiration(TimeSpan.FromMinutes(15)));
        // ↑ EXPIRE: 15 min tak koi harkat nahi? Entry khud nikal jayegi.
}

SizeLimit = godown ki hadd (bound). SlidingExpiration = miyaad (expire) — "sliding" matlab har istemal pe timer reset. Java duniya mein isi kaam ki mashhoor library Caffeine hai. Usool ek: jama karne wale har dabbe pe hadd aur miyaad likho.

STACK 3 • SQL SERVERRuko — yeh data memory mein hona bhi chahiye tha?

Ab woh sawal jo senior banata hai. Failed logins security ka data hai — bank mein regulator poochhega: "is account pe kab kab ghalat password laga?" Danish ka Map to restart pe sab bhool jata hai. Aisi cheez ki asli jagah memory nahi — database hai. Chalo table design karte hain:

CREATE TABLE FailedLogins (
    Id         BIGINT IDENTITY PRIMARY KEY,  -- IDENTITY = khud barhne wala number
    Cnic       VARCHAR(13)  NOT NULL,        -- CNIC: sirf digits/dash — VARCHAR kaafi
    AttemptAt  DATETIME2    NOT NULL
               DEFAULT SYSUTCDATETIME()      -- UTC waqt — Day 0 ka sabaq, ab amal mein
);
DEFAULT = agar INSERT mein yeh column na do to database khud yeh value daal dega. SYSUTCDATETIME() = abhi ka UTC waqt — server ka local timezone nahi, kyunke bank ke audit mein waqt ka EK sach hona chahiye (Zarr wale Rehan sahab ka sawal yaad hai?).

Record karna — ek line:

INSERT INTO FailedLogins (Cnic) VALUES (@cnic);   -- @cnic = parameter (Day 1 ka plan-cache dost!)

Check karna — "pichle 15 minute mein is CNIC ki kitni koshishein?":

SELECT COUNT(*)
FROM   FailedLogins
WHERE  Cnic = @cnic
  AND  AttemptAt > DATEADD(MINUTE, -15, SYSUTCDATETIME());
       -- DATEADD(MINUTE, -15, abhi) = "abhi se 15 minute pehle"

Aur ab dhyan se — table bhi to jama karti hai! Wohi Danish wala sawal yahan bhi banta hai: "niklegi kab?" SQL duniya mein "expire" ka naam hai retention job — ek scheduled safai:

DELETE FROM FailedLogins
WHERE AttemptAt < DATEADD(DAY, -30, SYSUTCDATETIME());  -- 30 din se purana = bahar (raat ka job)

To memory ya database? Trade seedha hai: memory ka check nanoseconds ka hai, DB ka round-trip milliseconds ka — magar DB restart pe bachta hai, audit deta hai, aur (agla section dekho) sab servers ko EK sach dikhata hai. Asli banks aksar dono rakhte hain: garam check cache mein, asal sach DB mein. (Redis naam ka ek beech ka kirdar bhi hai — Level 5 mein milega.)

DESIGN PATTERN • SINGLETONDanish ka static dar-asal ek pattern ka kachcha roop tha

Ab ek raaz: Danish jo karna chah raha tha — "poore program mein login-security ka EK hi markaz ho" — woh khwahish bilkul jaayez thi. Itni jaayez ke uska ek mashhoor design pattern hai: Singleton.

Design pattern = bar bar aane wale masle ka aazmaya hua, naam-diya-gaya hal. Naye nahi banane parte — pehchan'na seekhna hota hai. Jaise khaano ki recipes: "biryani" bolo to poora tareeqa zehen mein aa jata hai.

Singleton ka WHAT: aisi class jiska poore program mein sirf EK object ban sake, aur sab usi ko use karein:

public class LoginSecurity {
    private static final LoginSecurity INSTANCE = new LoginSecurity();
    private LoginSecurity() {}          // constructor private — bahar se "new" MANA
    public static LoginSecurity get() { return INSTANCE; }
}
// use: LoginSecurity.get().recordFailure(cnic);

WHY: jo cheez fitratan ek honi chahiye — configuration, connection pool, cache — uske 50 nuskhe na banein. WHEN NOT (khatra): Singleton ke andar badalne wala data rakho to wohi "global state" ka jurm hai jo aaj ke muqadme mein tha — har jagah se koi bhi badal sakta hai, tests mein ek doosre ka data ghusta hai. Aaj ka sach: Spring aur .NET Core mein tum Singleton haath se nahi likhte — framework ka DI container deta hai (services.AddSingleton<...>(); Spring ki har @Service default singleton hai). Kaise? — Level 3 mein poora hafta isi pe. Interview ka sona: "I know the Singleton pattern — and why it often turns into an anti-pattern (global mutable state), which is why we let the DI container manage lifetimes instead."

ARCHITECTUREDo servers — aur security chupke se toot gayi

Aakhri scene, aur aaj ka sab se maza-daar twist. TICH-KOLEY chal para. Traffic barha. Tumne doosra server laga diya, aage ek load balancer (traffic baantne wala chowkidar). Danish ka fix (IMemoryCache — shabash) har server pe chal raha hai. Rule: 10 failed attempts = account lock.

Ek raat ek attacker ne Fatima ke account pe 10 ghalat passwords maare. Account lock hua? Nahi. Dekho kyun:

                    ┌──▶ Server A ka cache: { attacker: 5 }   "abhi to 5 hi hain"
Attacker ──▶ LB ──▶─┤
                    └──▶ Server B ka cache: { attacker: 5 }   "abhi to 5 hi hain"

10 attempts the — magar KISI EK server ne 10 nahi dekhe. Hadd kabhi cross nahi hui.

Har server ki memory uski apni hai — load balancer ne attempts baant diye, aur security control bina kisi error ke, chupchaap, bekaar ho gaya. Yeh bug kisi log mein nahi aata. Isi liye architecture ka usool:

Jo state SAB servers ko ek jaisi dikhni chahiye (security counters, balances, sessions), woh kisi EK server ki memory mein nahi reh sakti — shared jagah chahiye: database (ya shared cache). Aur jab service apne paas aisi state rakhti hi nahi, usko stateless kehte hain — stateless services ko tum jitne marzi copies mein chala do, koi hisaab nahi bigarta. (Yeh lafz Week 3 mein HTTP ke saath phir milega — aaj bas pehchan.)

🔍 WHY • WHAT • HOW • WHEN — aaj ka concept 4 lines mein
WHY — Program ko do tarah ki memory chahiye: chhoti umar wali (method ke locals) aur lambi umar wali (objects jo method ke baad bhi chahiye).
WHAT — Stack = plates ka meenar (locals + parchiyan; method khatam = plate gone). Heap = godown (objects; parchi/reference se pahunchte ho).
HOW — Object tab tak zinda jab tak koi parchi us tak pahunchti hai; lawaris ko GC uthata hai; static parchi kabhi nahi phatti — us se bandhi cheez kabhi nahi marti.
WHEN — Jab bhi kuch jama karo (Map/List/cache), usi waqt poochho: "nikle ga kab?" Jawab nahi hai to leak likh rahe ho. Aur "memory chart sirf upar" dekho to restart nahi — heap mein dekho KYA barh raha hai.
🧠 Yaad rakhne ke 3 hooks
Plates ka meenar = stack (method khatam, plate gone) • Godown + parchi = heap + reference (parchi hai to zinda) • Static = amar parchi (kabhi nahi phatti — is se bandha jama hota gaya to leak) • Har dabbe pe hadd aur miyaad = bound + expire (Map ho, cache ho ya SQL table) • Do dukaanein, alag register = per-server memory pe shared hisaab kabhi nahi.

🎤 INTERVIEW DRILLAaj ka panel

MAIN QUESTION + English script (bol ke DO dafa, 60 sec)

"Explain stack vs heap — where do things live in a Java program?"

"The stack holds each method's local variables and references — every method call gets a frame, and when the method returns, the whole frame is discarded instantly, which is why stack allocation is so fast. Objects created with 'new' live on the heap, and what I hold on the stack is just a reference — an address pointing to them. An object stays alive as long as something reachable still references it; once nothing does, the garbage collector reclaims it. That's also why Java can still leak memory: if a static collection keeps referencing entries forever, they're never garbage — the classic leak is not lost memory, it's a forgotten reference."

🔑 Drill key — jawab likhne ke BAAD (trick ke naam ke saath)

C1: Plate pe sirf parchi (c) thi — asal object godown (heap) mein hai. return c ne parchi ki COPY caller ko de di (fatima), to object ki ek zinda parchi ab bhi maujood hai → object zinda. Trick: "plate pe object nahi, parchi thi".

C2: cust.getName() = "Ali". (1) parchi ke raste asal object badla — caller ko dikhta hai. (2) sirf method ki APNI parchi pe naya pata likha — caller ki parchi wahi purani. Java/C# reference ki copy pass karte hain. Trick: "parchi ki copy" — Day 0 pe yeh sawal (A-000-02) chhoot gaya tha; aaj band ho gaya.

K1: Stack pe rakhna/uthana = bas meenar ka sirra aage-peechhe karna, aur poori plate ek saath phenki jati hai — na dhoondhna, na safai. Cross: Sab kuch stack pe is liye nahi: (1) plate method ke saath MARTI hai — jo cheez baad mein chahiye (Fatima ka account) woh plate pe nahi reh sakti; (2) do methods/threads ko EK hi object share karna ho to woh kisi ek ki plate pe nahi ho sakta — godown chahiye.

K2: GC sirf lawaris (unreachable) cheezein uthata hai — jo cheez kisi zinda parchi (khaas kar static) se bandhi hai woh uske liye "qeemti saamaan" hai, kachra nahi. Leak = bhoola hua reference. Cross: Godown jitna bharta hai, GC utna bar bar aur utni lambi safai karta hai — magar utha kuch nahi pata; yeh bekaar mehnat CPU kha jati hai → requests slow. Crash to tab hai jab jagah bilkul khatam ho.

S1: SELECT COUNT(*) FROM FailedLogins WHERE Cnic = @cnic AND AttemptAt > DATEADD(MINUTE, -30, SYSUTCDATETIME()); — check karo: COUNT hai? Cnic parameter se? Waqt ki shart DATEADD se? Teeno hain to full marks (spelling maaf).

P1: Singleton = class jiska poore program mein sirf EK object ho, sab wohi use karein (config/pool/cache jaise shared cheezon ke liye). Sab se bara khatra: mutable global state — har jagah se badla ja sakta hai, tests aapas mein ulajh jate hain. Cross: framework ka DI container deta hai — .NET: services.AddSingleton<>(); Spring: har @Service/@Component by default singleton hai.

D1: (1) Size ki hadd (bound) — max entries; (2) Miyaad (expiry/TTL). Bonus senior line: "ready-made bounded cache (IMemoryCache/Caffeine) use karunga" + "memory metric pe alert — chart charhta dikhe to crash se pehle pata chale."

D2: Har server ka cache apna hai — LB ne 10 attempts 5/5 baant diye, kisi ek ne hadd cross hoti nahi dekhi. Trick: "do dukaanein, alag register". Asal fix: yeh counter shared jagah pe le jao jo sab servers ko ek dikhe — database (aaj wali FailedLogins table sahi jawab hai!) ya shared cache. Usool: shared hisaab kisi ek ki memory mein nahi rehta.

⭐ Senior differentiator

Jab sab "stack fast, heap slow" ratta sunayen, tum ek jumla aage jao: "The classic Java leak isn't lost memory — it's a forgotten reference, usually a static collection that only ever grows. That's why the first symptom is GC pressure and slowness, not the crash." Yeh jumla bolne wala banda 4 YOE ka nahi lagta.

🚩 Red flags — yeh KABHI mat kehna

"Java mein memory leak nahi ho sakti, GC hai na." — Aaj ka poora muqadma isi jumle ke khilaf gawahi hai.
"Objects stack pe bante hain." — Object godown mein, plate pe sirf parchi. (C# struct ki kahani alag hai — kal.)
"Restart kar do." — Danish ne 3 dafa kiya. Chart phir bhi upar hi gaya.
